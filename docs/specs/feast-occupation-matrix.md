# A Feast for Odin - Occupation parity matrix

Status: **parity audit, blocking**  
Scope: 2016 Classic base game, occupation cards 1-190  
Authoritative sources: `shared/src/feast/golden/occupations.json`, official `tmp/pdfs/odin-appendix.pdf` pages 2-12, official `tmp/pdfs/odin-rulebook.pdf` pages 21-22, and the current reducers in `shared/src/feast/actions.ts` and `shared/src/feast/state.ts`.

## Audit conclusion

The engine loads the name, deck, points, timing label, and appendix clarification for all 190 cards, but this is data parity, not rules parity. At this audit snapshot, only cards 1 and 2 have any card-number-specific runtime behavior, both through `feastActionSilverCost`. They are marked **PARTIAL** because there are no card-specific tests proving every included and excluded cost. Cards 3-190 are **MISSING** as executable rules.

The generic `use_occupation` API is not accepted as an implementation of any row below. It lets a client choose unrelated net-state operations, accepts an `acknowledge` operation as resolution, is callable before turn and pending-decision checks, and does not prove the printed trigger, payment, target, replacement, timing, or cardinality. In particular, any played card can currently grant arbitrary score adjustments or execute unrelated operations. Full parity requires the server to derive legal card decisions from the exact card definition and from a recorded game event; the client may choose only among those server-produced decisions.

## Required hook families

The compact hook codes used in the matrix mean:

| Code | Required structured engine responsibility |
|---|---|
| `PLAY` | Forced or optional on-play resolver, with the card's exact once-only clause. |
| `EVENT` | Typed before/after event provenance for an action, acquisition, placement, worker move, or phase step. |
| `PHASE` | Deterministic scheduler at the exact named phase boundary; simultaneous choices resolved in rules order. |
| `COST` | Composable cost discount, surcharge, alternate payment, minimum, and exclusion rules. |
| `GAIN` | Exact, server-computed conditional reward; never a client-supplied delta. |
| `XCHG` | Atomic validated payment/exchange transaction with exact inputs and outputs. |
| `UPGRADE` | Shape/color/step constrained goods transformation, including whether it counts as an upgrade action. |
| `REPLACE` | Replacement-effect layer that suppresses the replaced result and prevents double resolution. |
| `DICE` | Attempt context, roll count, modifiers, weapon/resource spending, success/failure, and refund hooks. |
| `WEAPON` | Draw/search/discard/return operations that preserve the finite weapon deck and substitutes. |
| `MOUNTAIN` | Arrow-end selection, item identity, strip identity, removal events, and aging events. |
| `SHIP` | Ship counts/types, berth limits, ore state, acquisition/use/build classification, sale, and emigration. |
| `ANIMAL` | Pregnant/non-pregnant stable counts, exact type equivalence, payment, and breeding classification. |
| `BOARD` | Board/house/exploration ownership, income snapshot, bonus production, and placed-vs-supply goods. |
| `PLACE` | Legal placement transaction plus provenance for board, house, stable, Banquet Table, and special replacement. |
| `ACTION` | Printed/occupation action identity, category, column, worker count, prerequisites, and copied-action context. |
| `WORKER` | Exact Vikings on an action space/column, active solo color, return/move/reuse, and Thing Square transitions. |
| `OCC` | Draw/play/discard occupation pipeline, fourth-column timing, and play-vs-draw replacements. |
| `SPECIAL` | Unique special-tile supply/ownership/placement and atomic movement from another board. |
| `FEAST` | Banquet Table history, horizontal/vertical exception, reward timing, extra feast, and penalties. |
| `BONUS` | Real board-scoped bonus resolution, including produced-count and replacement events. |
| `THRESH` | Continuously re-evaluated as-soon-as threshold with a durable one-shot latch. |
| `SCORE` | Server-computed scoring-time adjustment; no arbitrary VP/silver operation. |
| `OTHER` | Authorized other-player target selection and atomic cross-owner board replacement. |

## Blocking missing-hook register

This register groups the implementation gaps. Card numbers can appear in more than one group because the printed clauses compose.

- **Card contract and legality - all 1-190.** The server must map each card to exact clauses and reject any operation, timing, target, amount, or action identity not declared by those clauses. Cards 1-2 have partial direct hooks; no card currently has complete clause-level legality and verification.
- **On-play resolution - 5, 10-14, 16-27, 29-35, 43, 60-87, 183-184, plus the on-play halves of 11-12, 13, 15, and 188.** Playing the card must immediately queue its exact optional/mandatory decision and block continuation until resolved or legally declined.
- **Anytime windows - 39-40, 44-59, 187.** The window must be legal during the rulebook's anytime periods but must not interrupt an atomic reducer decision or retroactively alter an already snapshotted phase.
- **Phase scheduling - 5, 7-9, 20-21, 35, 38, 41-42, 49, 91-97, 157-162, 165, 167, 169, 171-173, 177, 186, 189.** Each needs an exact pre/start/during/after/scoring boundary. Generic `phase` operations are insufficient; the current bonus operation is a no-op.
- **Replacement and alternate-result layer - 38, 46, 49, 53, 64, 72, 77-79, 83-85, 87, 93, 99, 102, 105, 107, 110, 123, 136-139, 147, 150, 152, 169-170, 177, 185-186.** Replaced effects must not also pay the original result, and classification text such as "not a Ship Building action" must affect downstream triggers.
- **Dice/attempt context - 4, 11, 13, 77, 85, 89-90, 104, 126-146, 153, 163.** The engine needs per-roll modifiers, expanded roll count, exact spend/refund records, deliberate failure, result splitting, and success/failure events.
- **Placement provenance - 9, 40, 42, 70-71, 98, 100-101, 157-160, 172, 184, 186.** A net inventory edit cannot prove where or when a piece was placed, moved, or replaced.
- **Worker/action-space mutation - 60, 76-77, 91, 106, 108, 130, 151, 153-154, 155-156, 161-164, 166, 168.** The current occupation operation union cannot move Vikings, reopen a space, reduce worker cost, or prove a column/active-color count.
- **Other-player interaction - 70-71.** These require target selection, unique-tile ownership transfer from any player's committed placement, and exact replacement pieces in the vacated cells.
- **Threshold latches - 178-182.** The trigger must fire at the first qualifying state transition after play, exactly once, with capacity/income predicates evaluated at that moment.
- **Scoring hook - 189.** Exploration count must award silver during scoring before the final total is computed. Generic arbitrary `score` adjustments are both too broad and the wrong resource/timing.
- **Finite supply/deck conservation - 3, 11-13, 28, 32, 56, 65, 68-71, 73, 87, 104, 129-130, 135, 142, 158, 166, 178, 184, 187.** Search/draw/discard/return/special/ship operations must preserve component counts and honor unavailability.

## Verification gates for claiming occupation parity

All of the following are required in addition to the per-row tests:

1. **Registry completeness:** exactly 190 numbered definitions, every appendix clause represented, every clause assigned a typed trigger, predicate, choice cardinality, operation, classification, and once/per-action/per-round latch where applicable. A metadata-only clarification string does not satisfy this gate.
2. **No client-authored effects:** remove or hard-disable the unrestricted public `use_occupation` net-state operation path. Every occupation action sent by a client must reference a live server-issued decision and choose only declared option IDs/amounts.
3. **Event provenance:** reducers must emit immutable typed facts for action start/end, category/source, payment, gross gain, acquisition, placement, worker movement, ship use/build/acquisition, attempt roll/spend/result, and exact phase boundary. Occupations react to facts, not inferred final inventory.
4. **Replacement ordering:** a deterministic replacement stack must prove that the original effect is suppressed exactly once, composed discounts respect minima/exclusions, and downstream triggers see the official classification after replacement.
5. **Atomicity:** a card decision either validates and commits in full or changes nothing. This includes multi-resource payments, cross-player tile moves, ship+ore creation, worker returns, extra actions, and phase inserts.
6. **Finite component conservation:** seeded tests must conserve the 47-card weapon set, unique special tiles, ship/house supplies, exploration ownership, goods placements, workers by color, and occupation draw/discard/played zones across every card path.
7. **Timing determinism:** replay tests must cover before/start/during/after boundaries, nested occupation actions, fourth-column card play before/after the printed action, solo active colors, simultaneous phase triggers, and save/load while a card decision is pending.
8. **Information boundaries:** private occupation hands and private choices remain visible only to their seat; public played cards, cross-player targets, and resulting events are visible and auditable.
9. **UI derivation:** the occupation UI must render choices from the same server decision schema used by bots and validation. It must show the card art, exact trigger, payment, result, legal targets, disabled reasons, and a visual before/after preview; no free-form note or manual inventory editor may appear.
10. **Exhaustive suite:** at least one positive and every meaningful boundary/negative path in each row above, plus pairwise composition tests for interacting cards. The suite must fail if any card is replaced by `acknowledge`, an arbitrary delta, a wrong event source, or an off-by-one timing window.
